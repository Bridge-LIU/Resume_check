2. config/settings.json の平文保存

lib/storage.ts
問題: ファイル ACL なしで API キーが平文 JSON。バックアップ・エディタ拡張・誤コピーで容易に流出。
暫定対応: 書き込み時に mode: 0o600（本人のみ rw）を強制。 WSL/Linux/macOS では実効的に保護、Windows は OS が POSIX bit を無視するので限定的（環境変数誘導を併用）。
本格対応（未実施・要相談）: keytar / Windows DPAPI で OS Credential Store に格納。
Before
export function saveSettings(s: Settings): void {
  fs.mkdirSync(path.dirname(SETTINGS_PATH),
    { recursive: true });
  fs.writeFileSync(
    SETTINGS_PATH,
    JSON.stringify(s, null, 2),
    "utf-8"
  );
}
After
export function saveSettings(s: Settings): void {
  fs.mkdirSync(path.dirname(SETTINGS_PATH),
    { recursive: true });
  // 本人のみ rw（WSL/Linux/macOS 有効、Windows は noop）
  fs.writeFileSync(SETTINGS_PATH,
    JSON.stringify(s, null, 2),
    { encoding: "utf-8", mode: 0o600 });
  try { fs.chmodSync(SETTINGS_PATH, 0o600); }
  catch { /* Windows noop */ }
}

3b. LlmCallError がレスポンス本文をそのまま UI に伝播

lib/llm/types.ts
問題: await res.text() を error message にそのまま連結していたため、 上流が URL や header をエコーする HTML ページを返した場合に Bearer / sk- / AIza 等が クライアント側 <div role="alert"> に出る可能性。 対応: redactSecrets() で正規表現マスク + 500 字に切り詰め。
Before
export class LlmCallError extends Error {
  constructor(
    public provider: ProviderId,
    public status: number,
    message: string,
  ) {
    super(
      `${provider} API error ${status}: ${message}`
    );
    this.name = "LlmCallError";
  }
}
After
const SECRET_PATTERNS: RegExp[] = [
  /sk-(?:ant|proj|live|test)[-_][A-Za-z0-9_-]{8,}/g,
  /sk-[A-Za-z0-9_-]{20,}/g,
  /AIza[0-9A-Za-z_-]{20,}/g,
  /Bearer\s+[A-Za-z0-9._-]{16,}/gi,
  /key=[A-Za-z0-9_-]{16,}/g,
  /x-api-key:\s*[^\s,;]+/gi,
  /x-goog-api-key:\s*[^\s,;]+/gi,
];

export function redactSecrets(input: string): string {
  let out = input;
  for (const pat of SECRET_PATTERNS)
    out = out.replace(pat, "[REDACTED]");
  if (out.length > 500) out = out.slice(0, 500) + "…";
  return out;
}

export class LlmCallError extends Error {
  constructor(provider, status, message) {
    super(`${provider} API error ${status}: ` +
          redactSecrets(message));
    …
  }
}

6. /api/backup に Origin チェック無し（CSRF 余地）

app/api/backup/route.ts
問題: 認証なしのローカル運用なので、ブラウザ拡張や別タブからの fetch('http://localhost:3939/api/backup', {method:'DELETE', …}) で破壊操作を実行できる余地があった。 対応: POST / DELETE に ensureLocalOrigin() を入れ、 localhost:3939 / 127.0.0.1:3939 / [::1]:3939 以外を 403 で拒否。
新規追加
function ensureLocalOrigin(req: Request): void {
  const origin = req.headers.get("origin")
    ?? req.headers.get("referer") ?? "";
  if (!origin) return; // same-origin は Origin が無い場合あり

  let host = "";
  try { host = new URL(origin).host; }
  catch {
    throw new ApiError("FORBIDDEN_ORIGIN",
      "不正な Origin ヘッダです", 403);
  }
  const allowed = new Set([
    "localhost:3939",
    "127.0.0.1:3939",
    "[::1]:3939",
  ]);
  if (!allowed.has(host)) {
    throw new ApiError("FORBIDDEN_ORIGIN",
      "ローカル以外からの破壊的操作は許可されていません", 403);
  }
}

export async function POST(req)   { ensureLocalOrigin(req); … }
export async function DELETE(req) { ensureLocalOrigin(req); … }
// GET は据え置き（listBackups は影響小）
