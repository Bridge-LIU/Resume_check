B 案：履歴書を補助情報として渡す

提案
入力 = ④凍結条件 + ②履歴書要約（参考） + ⑥議事録。 system に「議事録優先・根拠は議事録引用」を追加して halo を抑止。
SYSTEMEVAL_SYSTEM_PROMPT（追記あり）
あなたは採用評価の専門家です。BARS（行動基準評価）で厳正に採点し、説明文や前置きなしに、指定スキーマの JSON のみを出力してください。総合スコアは軸スコア × 軸重みの加重平均（重みが未指定なら単純平均）。合否は提示された評価条件の「合格ライン」以上=合格、「普通ライン」以上=普通、未満=不合格（数値はユーザーメッセージの評価条件 JSON から読み取り、自分で仮定しないこと）。
採点の根拠は必ず「⑥ 面談議事録」からの引用を主とすること。「② 候補者背景」は議事録に直接表れない技術スキル・資格・年数などを補う参考情報であり、これ単独を根拠にしてはいけない（書類 halo を避ける）。背景情報と議事録が矛盾する場合は、議事録を優先する。

A → B で変わるところまとめ

1. system に halo 抑止文を追加（議事録優先・背景単独で加点禁止・矛盾時は議事録優先）
2. user に「# 候補者背景（②要約・参考情報）」セクションを追加（履歴書要約があるときだけ）
3. コード変更箇所: app/sessions/[id]/actions.ts の evaluateInterviewApiAction（API）と buildEvaluationPromptAction（貼付）の 2 関数のみ
4. UI 変更なし。⑧評価セクションの見た目は変わらず、内部プロンプトだけが豊かになる
5. 影響範囲: 既存セッションの評価結果は変わらない（再評価しない限り）。 新規 / 再評価のみ B 案プロンプトが使われる

期待効果

  • 「履歴書には書いてあるが議事録で深掘りされなかった」項目を「未確認」として明示できる
  • 未経験可 = true 案件で「条件②（必須項目）」の充足判定が背景込みで判断できる
  • 軸別根拠の質が上がる（背景と議事録の整合チェックが入る）

残るリスク

  • halo 抑止プロンプトが効かないモデル（弱いモデル）だと、書類で +0.3 〜 0.5 上振れする可能性
  • 履歴書が長文だとトークン消費が増える（要約は800字想定なので軽微）
  • 履歴書が未入力のセッションでは A 案と同じ挙動になる（フォールバック）
