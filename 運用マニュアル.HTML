面談AI評価ツール

採用面談 1 件あたりの「候補者要約 → 条件凍結 → 質問づくり → 面談内容 → 評価」をローカル PC で完結させるツールです。

このマニュアルについて

左側のメニューで章を切り替えます。各章は 独立したページ として表示され、右側の「このページ内」はその章の小節だけを案内します。右上の 🔍 または Ctrl+K で全ページから横断検索できます。

主な機能

  • 面談ごとに専用 URL を作って、関連情報を 1 ヶ所に集約
  • 履歴書要約・面談内容・評価結果を 貼付 または API の 2 モードで保存
  • 役割（Dev / ITSupport / NW / PMO / Server の 5 種を同梱、追加は /master で自由）ごとに別々の評価条件
  • 面談時点の条件を 凍結 → 後でマスタを変えても過去面談に影響しない
  • 過去面談を 匿名サマリとして集計・分析
  • Excel ファイル（マスタ / 面談者一覧）を自動更新
  • 暗号化バックアップを世代管理

動作モード

各セクション（① 面談者情報 / ③ 質問 / ④ 面談内容 / ⑤ 評価）は 貼付モード と API モード をボタンで切り替えて使えます。

  • 貼付モード（既定）: AI 要約・質問生成・評価を Claude / ChatGPT 等の AI チャット側で行い、結果をコピー → このツールに貼付
  • API モード: /settings で登録した Provider（Anthropic / OpenAI / Google）を直接呼び出し、要約・生成・評価を自動化。使用状況は /cost で集計

API キー未登録の状態でも貼付モードだけで業務が完結します。API モードを有効にするには 設定 参照。

起動方法

プロジェクトフォルダの start-app.bat をダブルクリックするだけです。

初回

  1. Node.js 20 以上 をインストール（nodejs.org から、インストーラの「Add to PATH」にチェック）
  2. プロジェクトフォルダを PC の好きな場所に置く
  3. start-app.bat をダブルクリック
  4. 初回だけ npm install（3〜5 分）と next build（1〜2 分）が走る
  5. ブラウザが自動で開き http://localhost:3939 が表示される

2 回目以降

start-app.bat ダブルクリック → 数秒でブラウザが開きます。

停止

起動時に開いた コマンドプロンプトで Ctrl+C → Y → Enter。

ブラウザを閉じるだけでは停止しません。コマンドプロンプトでの操作が必要です。

基本フロー：全体像

1 件の面談は 新規面談→①→②→③→④→⑤ の順で進めます。

セッション本体（全セクション）

ステータスの自動更新

各ステップで保存すると、対応するステータスに自動で進みます：

  • 新規作成 → 編集中
  • ② 条件凍結 → 質問公開
  • ④ 面談内容保存 → 面談済
  • ⑤ 評価保存 → 評価済

セクション内ナビ（サイドバー）

セッション本体の左側には ①〜⑤ の縦型ナビが常時追従表示されます。

  • クリックでその工程まで即ジャンプ（スムーススクロール）
  • スクロール位置で現在のセクションを自動ハイライト
  • 各項目の末尾に ✓（入力済） / ○（未入力） のマークが付き、進捗が一目でわかる

共通セクションヘッダー

各セクション（①③④⑤）の先頭には 緑の左ボーダー付きヘッダーバーがあります。保存済みの工程は 保存済 chip が並び、未着手の工程はグレーのままになります。

② 条件凍結だけは専用のヘッダーで、条件チップ群と「修正」ボタンが並びます。

新規面談を作る 編集中

面談ごとに専用の URL を作るのが最初のステップです。

作成手順

  1. 画面右上の 「+ 新規面談」 をクリック
  2. 候補者の 氏名 と、応募 役割（Dev / ITSupport / NW / PMO / Server の 5 種、/master で追加可能）を入力
  3. 「作成して開く →」 をクリック
  4. 自動的にセッションページ /sessions/<ID> に遷移

新規作成画面

セッション ID について

ID は YYYYMMDD_HHMMSS_<氏名>_<役割> 形式で自動生成。秒まで含むので同一分内の連続作成でも ID 衝突しません。

役割を間違えても 複製機能 で後から変更できます。

① 面談者情報 編集中

履歴書を AI で要約して、結果をこのエリアに保存します。API モード（直接ファイルアップロード）と 貼付モード（AI チャット経由）の 2 通り。

API モードの手順（推奨）

キー登録済みなら 1 クリックで要約完了。履歴書 + 職務経歴書のような 2 種類のファイルもまとめて 1 要約に統合できます。

  1. ① エリア右上の モード切替 を API に
  2. Provider / モデルを選択（未選択なら /settings の既定を使用）
  3. 「📎 履歴書をアップ」 をクリック → PDF / Word / Excel を添付（最大 2 ファイル）
  4. 「AI で要約」 ボタンをクリック
  5. 数秒〜30 秒で要約が「要約」枠に自動投入 → フォーカス外で 自動保存

貼付モードの手順

API キー無しでも AI チャット（Claude / ChatGPT / Gemini）経由で同等の結果を貼付できます。

  1. ① エリア上の 「📋 コピー」 ボタンで履歴書要約プロンプトをクリップボードへ
  2. AI チャットに貼付 → 履歴書ファイルを添付
  3. AI 回答をコピー（Ctrl+A → Ctrl+C）
  4. ① エリアの「要約」テキスト枠に貼付
  5. フォーカスを外すと 自動保存

API モード詳細：ファイルアップロード仕様

  • 対応形式: PDF / Word (.doc / .docx) / Excel (.xlsx / .xls)
  • 1 ファイル上限: 3.7 MB（Server Action の 5 MB body 制限に収まる base64 換算）
  • ファイル数上限: 2 ファイル（既定「履歴書」+「職務経歴書」を想定）
  • UI: 1 個目を選ぶと積み上げリスト表示、追加ボタンが「📎 履歴書をアップ（2/2）」に変わり、2 個目で消える
  • 取消: 各ファイル行の 「取消」 ボタンで個別除外
  • 要約時: 複数ファイル時は AI プロンプトに #添付 1 / #添付 2 の見出しが付き、AI が区別して統合要約

旧 .xls ファイルはブラウザ内で自動的に .xlsx に変換されてから送信されます（サーバー側では扱えないため）。

要約テキストの Excel 自動分割

要約本文に「経歴サマリ：」「主要スキル：」「強み：」の見出しを入れると、Excel エクスポート時に自動で 3 列に分割されます。API モードのプロンプトは既定でこの形式で返します。

② 条件を凍結する → 質問公開

求人情報の条件と評価軸をこの面談に固定します。

何をするのか

求人情報の「求める人材条件」と「評価条件」を、この面談に スナップショットとして固定します。後でマスタを変えてもこの面談には影響しません。

手順

  1. ② エリアで「現在の条件」を確認
  2. 下部の 「この内容で凍結する」 ボタンをクリック
  3. 確認ダイアログで「OK」
  4. ステータスが 質問公開 に変わり、③ 質問エリアが触れるようになる

「凍結」＝今の評価条件を写真のように コピーして保存すること。

あとでマスタの重み・軸・合格ラインを変えても、この面談は当時の条件で評価され続けます。

例：マスタで「NW はコミュニケーション力の重みを 3 → 2 に下げる」設定にしていれば、その面談のコミュニケーション力の重みは 2 で固定。
あとから NW の重みを 3 に戻しても、この面談だけは 4 で計算されます。

凍結後に条件を修正したい場合

凍結済ヘッダー右側の 「修正」 ボタンで編集モードに入れます。人材条件・評価軸・重み・合格ラインを直接書き換え、下部の 「この内容で再凍結する」 ボタンで凍結条件が上書きされます（マスタは変わりません）。誤入力・軸抜けを直したいときに利用してください。

⑤ 評価保存後は凍結条件を変えると 過去スコアの意味が変わる ため、原則として評価前に完結させます。

③ 質問を貼る 質問公開

候補者の経歴と凍結条件から面談質問を生成し、③ エリアに保存します。API モード（AI が自動生成）と 貼付モード（AI チャット経由）の 2 通り。

API モードの手順（推奨）

キー登録済みなら 1 クリックで質問リストが生成されます。

  1. ③ エリア右上の モード切替 を API に
  2. Provider / モデルを選択
  3. 「AI で質問生成」 ボタンをクリック
  4. 候補者要約 + 凍結条件で API を呼び出し（数秒〜30 秒）
  5. 「人間性」「技術」の 2 枠に自動振り分けされ、下部にカードプレビュー表示 → 「保存」

使用トークンは /cost に記録されます。事前に 設定 で API キーの登録が必要です。

貼付モードの手順

API キー無しでも AI チャット経由で質問を生成できます。

  1. ③ エリア上の 「📋 コピー」 ボタンで質問生成プロンプトをクリップボードへ
  2. AI チャットに貼付 → 15 問前後を生成
  3. AI 回答をコピーして「人間性」枠 / 「技術」枠に振り分けて貼付
    • AI が ## 人間性 / ## 技術 見出しを付けていれば 1 枠に全文貼付で自動振り分け
  4. 「保存」 → 下の「構造化プレビュー」でカード表示

質問の形式

質問テキスト内で以下の接頭辞・キーワードが使えます（AI に生成させる時点でこの形式にしておくとカードに反映されます）:

  • ⭐（行頭）：必須質問マーク。カードに ⭐ が表示される
  • Q1. / T1. のような Q / T + 番号：質問見出し
  • 狙い: 〜：質問の意図
  • 解答例: 〜：参考解答

編集はテキスト枠内で直接行います（並べ替え・削除も文字ベース）。保存後のカードは表示専用です。

質問数の目安

既定は 人間性 7 問 + 技術 8 問。カウンタが目標値のとき緑、超過は黄で色付けされます。プロンプトの文中の数字を書き換えれば任意の件数で生成できます。

コピー系ボタン

  • 本文コピー：狙い / 解答例を除いた質問本文のみをコピー（面談本番の司会用）
  • 全文コピー：狙い / 解答例を含めて全部コピー（面談官への配布用）

④ 面談内容を貼る → 面談済

面談本番の文字起こしや手書きメモを ④ エリアに貼り付けます（モード切替なし・貼付運用のみ）。

手順

  1. 面談本番は Teams など別ツールで実施 → 文字起こし
  2. 文字起こし（または手書きメモ）を全選択コピー
  3. ④ エリアに貼付 → 「保存」
  4. ステータスが 面談済 に変わる

長すぎる文字起こしは事前に ChatGPT / Claude で要点抽出してから貼るのがおすすめ。テキスト内に文字数の上限は約 200 万文字なので、通常の 60 分面談なら余裕で収まります。

面談内容の例（貼付イメージ）

フォーマットは自由（Teams 文字起こしそのまま／手書きメモ／整形済みどれでも可）。上限は約 200 万文字なので、通常の 60 分面談なら余裕で収まります。参考までに整形版の例：

日時: 2026-06-30(月) 14:00-15:00
面談官: 採用担当 A / 現場リード B
候補者: 測試 太郎（Test 役割応募）
形式: Teams（録画あり）

質問番号（Q1 / T1 など）を残しておくと、⑤ 評価プロンプトを流したときに AI が該当質問と紐付けた根拠を返しやすくなります。

⑤ 評価を貼る → 評価済

面談内容と凍結条件から評価を確定します。API モード（AI が自動評価）と 貼付モード（JSON 手貼付）の 2 通り。

API モードの手順（推奨）

キー登録済みなら 1 クリックで評価が完了します。

  1. ⑤ エリア右上の モード切替 を API に
  2. Provider / モデルを選択（「厳格モード」で上位モデルに切替も可）
  3. 「AI で評価」 ボタンをクリック
  4. 面談内容 + 凍結条件で API を呼び出し（30 秒〜数分）
  5. 各軸スコア・根拠・良い点・懸念点が自動投入 → 「保存」
  6. 採否（採用 / 不採用 / 未確定）を選ぶ → ステータス 評価済

厳格モードは通常より上位の推論モデル（Claude Opus / GPT o1 等）を使うため精度は高いが料金も高くなります。/cost でリアルタイム使用状況を確認できます。

貼付モードの手順

API キー無しでも AI チャット経由で評価 JSON を作って貼付できます。

  1. ⑤ エリアの「評価結果 JSON を貼り付ける」を開き、「📋 コピー」 で評価プロンプトをクリップボードへ
  2. AI チャットに貼付 → 面談内容 + 凍結条件から JSON 形式で評価結果を生成
  3. AI が返す JSON を評価 JSON 枠に貼付 → 「保存」
    • AI が ```json フェンスで囲んでいたら囲みを除いてから貼付（素の JSON のみ受け付け）
  4. 各軸スコア・総合スコア・合否（合格 / 普通 / 不合格）が自動表示
  5. 採否を選ぶ → ステータス 評価済

注意

凍結条件を変えるとスコアの意味が変わるため、評価後の凍結条件変更は基本不可です。

編集不可 フラグ

役割単位で 「編集不可」チェックボックスを ON にすると、その役割で作成された新規面談は以下のように動作します。

  • 新規セッション作成時に ④ 求める人材条件が自動で凍結される（conditions_snapshot.json を即時生成）
  • セッション画面の 「修正」ボタンが無効化され、条件を書き換えできない
  • ホバーで「マスタで編集不可に設定されているため修正できません」と理由が表示される

用途: 契約先が決めた評価条件を面談官が勝手に変えないよう固定したい場合、監査観点で凍結条件の一貫性を保証したい場合など。

編集不可の役割は 🔒 アイコンが表示されます（一覧表・新規面談の役割選択・セッションの ② ヘッダ）。

画面

データ保存先（dataRoot）

画面

バックアップ

/settings 下部の「バックアップ」セクションで管理します。

作成

  1. お好きなパスワードを入力（バックアップの中身を守るため）
  2. 「今すぐバックアップ」 ボタンをクリック
  3. これで完了。バックアップファイルが 1 つ作られます

面談データとマスタの中身をまるごと 1 ファイルにまとめて、入力したパスワードで暗号化して保存します。暗号化は AES-256-GCM、パスワードからの鍵導出は PBKDF2-SHA256（200,000 回反復）。パスワード無しの平文バックアップは作成できません。

パスワードの管理

パスワードを紛失すると 復元不可 です。パスワード管理ツールに保存推奨。

世代管理

  • 保存日数：既定 90 日（0 = 自動削除しない）
  • 世代上限：既定 0 = 無制限
  • 「世代を整理」ボタンで手動掃除可能

復元

復元経路は 2 通りです：

  • 一覧から復元: このアプリで作ったバックアップ一覧の各行の「復元」ボタン
  • 外部ファイルから復元: 別 PC / USB / 他人から受け取った .enc.tar.gz をアップロードして復元（上部の「📎 外部ファイルから復元」ボタン）

いずれも同じ preview → confirm → restore フローで安全に処理されます。

一覧から復元

設定 > バックアップ一覧の各行の 「復元」 ボタンから、選択したバックアップの内容で master/ ・ sessions/ ・ settings.json を復元できます。

  1. 復元したいバックアップ行の 「復元」 ボタンをクリック
  2. 作成時に設定した 暗号化パスワードを入力（上のパスワード欄が空の場合はダイアログで確認）
  3. プレビューが表示される（アーカイブ内の master ファイル数 / セッション数 / 上書きされるセッション数 / 追加される / 消滅する内訳）
  4. 内容を確認して 「復元する」 をクリック
  5. 復元前の現行データは自動的に data/_restore_snapshots/<タイムスタンプ>/ に退避されます（事故時のロールバック用）

プレビューは復号のみを行い、ファイルシステムには一切書き込みません。パスワードが間違っている場合はこの段階で失敗します（現行データは無傷）。

復元は破壊的操作です。実行前に「今すぐバックアップ」で作業中の変更を保存してください。パスワード違い・ファイル破損・不正パスは全て自動 reject されます。

外部ファイルから復元

別 PC で作ったバックアップや、他人から受け取った .enc.tar.gz をアップロードして復元できます。

  1. 設定 > バックアップ セクション上部の 「📎 外部ファイルから復元」 ボタンをクリック
  2. ファイル選択ダイアログで .enc.tar.gz を選択
  3. 選択したファイルは自動的に data/_backups/ に取り込まれ（新しいタイムスタンプで命名）、一覧に追加される
  4. 続けてパスワード入力 → プレビュー → 確認ダイアログが表示される
  5. 「復元する」で実復元

アップロード上限は 500 MB。元ファイル名は使わず、backup-YYYYMMDD-HHmmss.enc.tar.gz に自動リネーム。

CLI 経由の復元（緊急・オフライン時）

アプリが起動できない場合は、付属の CLI で復号 → 手動配置が可能です。

  1. アプリを停止（コマンドプロンプトで Ctrl+C）
  2. 現在の data\master\ と data\sessions\ を別フォルダにコピーして退避
  3. 復号スクリプトを実行:
    node scripts/decrypt-backup.mjs data\_backups\backup-YYYYMMDD-HHmm.enc.tar.gz restored\
  4. プロンプトに従ってパスワードを入力（Enter で確定）
  5. restored\master\ と restored\sessions\ を data\ 直下へ配置
  6. start.bat で再起動

スクリプトは復号だけを行い、既存データは書き換えません。事故防止のため出力先に既に master/ か sessions/ があると上書きせず終了します。

システムバージョン & 更新

/settings 画面下部の「システムバージョン」カードから、GitHub Releases 経由でワンクリックで最新版に更新できます。

更新をチェックする

  1. /settings を開き、下部の「システムバージョン」カードを確認
  2. 「更新をチェック」ボタンを押す
  3. 数秒待つと以下のいずれかが表示されます:
    • ✓ 最新版を使用中：更新は不要
    • 🎉 新しいバージョンが利用可能：v○.○.○ → v○.○.○ + 更新内容が緑色の枠に表示
    • チェックに失敗しました：ネットワーク不通 or Release が公開されていない

チェックは GitHub Releases API を叩いて最新公開版と比較します。認証は不要ですが、GitHub のレート制限（1 時間 60 回）が適用されます。

更新を適用する

新版がある状態で 「🚀 今すぐ更新」 ボタンを押すと、モーダルが開いて自動で以下 3 段階が実行されます:

  1. [1/3] ダウンロード：新版の ZIP を data/.update/staging/ に取得（50-80MB、~30秒）
  2. [2/3] バックアップ：現行版のファイルを .backup/vX.Y.Z/ に瞬時 move（データは移動しません）
  3. [3/3] インストール & 再起動：新版ファイルを配置 → start.bat 経由で新版起動 → バージョン確認

モーダル下部の cmd ログが黒背景でリアルタイム表示されるので、進捗を目視で確認できます。完了時に「✅ 更新完了 vX.Y.Z」トーストが出て、ブラウザが自動でリロードします。

更新中は cmd ウィンドウが新しく開き、既存のサーバーは一度停止します。更新中はブラウザ・cmd ウィンドウを閉じないでください。閉じても復旧できますが手動操作が必要になります。

失敗時の自動 rollback

  1. モーダルが「⚠️ 更新失敗、v○.○.○ に回滾中」表示に切り替わる
  2. updater.bat が .backup/ から旧ファイルを move で戻す（数十秒）
  3. 旧版の start.bat が再起動
  4. ブラウザが自動リロードして「回滾完了」表示 → 元通り使える

Rollback 完了後、失敗の原因は cmd ログに残ります。ネットワーク不通なら時間をおいて再試行、それ以外の場合は開発者に連絡してください。

手動復旧（restore.bat）

PC の電源断・ブルースクリーン等で updater.bat 自体が異常終了した場合、以下の緊急復旧手段があります:

  1. プロジェクト直下の scripts/restore.bat をダブルクリック
  2. 「利用可能なバックアップ」一覧から復元したいバージョン（通常は直近の v0.1.0）を選択
  3. 確認後、backup から旧ファイルを戻して復旧
  4. start.bat で通常起動

バックアップの世代管理

更新のたびに .backup/ 下に旧版のコピーが 1 世代残ります。次の更新開始時に古い世代は自動削除されるため、通常はディスク容量を意識する必要はありません。

  • 保持世代数：1（直近の旧版のみ）
  • 1 世代あたりのサイズ：数百 MB 程度（node_modules は含まない設計）
  • 手動で .backup/ を削除しても問題ありませんが、削除後は rollback / restore.bat が使えなくなります

更新で保持されるもの

  • data/ フォルダ：面談セッション・マスタ・設定・分析データ — すべて保持
  • settings.json の API キー・保存期間設定など
  • バックアップ ZIP（data/_backups/）

更新で書き換えられるのは .next/ public/ package.json start.bat scripts/ のみ。あなたのデータには一切触れません。

分析画面

/analytics で過去面談を匿名集計。「次の面談で何を重点的に聞くか」を決めるためのダッシュボードです。

画面

スクリーンショットは旧レイアウトのものです。実際のダッシュボードは下記 8 段構成にリニューアル済み。

画面構成（上から順）

#セクション答える問いポイント
1Hero このページは何？ ページの目的説明 + 全期間合格率を大字で表示
2コア指標（KPI 4 枚） 今の全体像は？ 前月比でどうか？ 合格率 / 平均総合スコア / 面談数 / 役割数 の 4 KPI。各カードに 前期比 ▲▼ と 6 月分の sparkline を並記
3インサイト（自動生成） データから何を読み取ればいい？ 3 枚のカードが自動で洞察を提示：
①「合否を分ける最重要軸」（合格者 / 不合格者 平均差 Δ が最大の軸）
②「合格率トップ役割」
③「要注意」（合格率急落 or 最低役割）
4月別トレンド 時間とともにどう変わっている？ コンパクト表：月 / 件数 / 合格率（色階） / 内訳分布（合格 · 普通 · 不合格 の内联 3 色バー、幅 = 最多月比）
5軸別分析 どの評価軸が合否を左右している？ 6 小軸を 「人間性（amber）/ 技術力（indigo）」帯でグループ化。各軸で合格者 vs 不合格者 の平均を 2 本並列バーで対比、右端に差分 Δ を表示（Δ 大 = 決定要因）
6役割別 どの役割の合格率が高い / 低い？ カード grid。合格率 % を大字（緑 / 黄 / 赤）+ 件数 + 平均 + 底部に pass/mid/fail 3 色分布条
7不採用理由の内訳 なぜ落ちているか？ セッションの「不採用理由」チェック（⑤ 評価で選択）を集計。未記入件数もバッジで明示。過去データで未記入のセッションは「未記入」バケットに入る
8明細 個別のデータを深掘りしたい 匿名 1 件ずつ。列は closedAt · 役割 · 6 小軸（大分類バンド付き）· 総合 · 合否 · 採否。全列 sort、カテゴリ列は filter 可、数値列は sort のみ。行 hover で走光 ring 表示

軸別分析の読み方

各軸に 合格者平均バー（緑）と 不合格者平均バー（赤）が並びます。右端の Δ 値が大きいほど、その軸のスコア差が合否を左右していると解釈できます。

Δ ≥ 1.5 で primary 色ハイライト表示。「次の面談ではこの軸の質問を厚めに」の判断材料に使ってください。合格者 or 不合格者のどちらか 0 件の軸は Δ 計算不可のため「—」表示。

月別トレンドの読み方

  • 合格率列：60% 以上 = 緑、40〜60% = 黄、40% 未満 = 赤で色階表示
  • 分布列：バーの幅は「最多件数月に対する相対値」で描画（絶対件数比較用）。3 色内訳は「合格 / 普通 / 不合格」の順で左詰め
  • closedAt が入っていないセッションは「不明」月として表末に固まる

明細の使い方

  • 列ヘッダのラベルクリックで sort 切替（asc → desc → 解除）
  • フィルタアイコン（closedAt · 役割 · 合否 · 採否 の 4 列のみ表示）で値ごとに絞り込み
  • 数値列（総合 · 6 小軸）は連続値のため filter アイコンなし、sort のみ
  • 大分類バンドは 人間性（amber） → 技術力（indigo） → 分類なし の順に並ぶ（マスタで軸を追加した場合も自動）
  • 行 hover で走光 ring（primary 色）+ glow shadow が出て、視線を追いやすくする
  • 数値の色階：総合スコアは ≥4 emerald / ≥3 通常 / <3 rose。小軸は ≥4 淡 emerald / <3 淡 rose で強弱点を強調

データが 0 件のとき

匿名サマリがまだ書き出されていない場合、ダッシュボードは表示されず「保存期間スイープで書き出されるまで待つ」旨のメッセージが出ます。ただし 不採用理由の内訳は生の SessionMeta から即座に集計されるため、匿名サマリ 0 件でも表示されます。

プライバシー

ダッシュボードのデータソースは 氏名・履歴書・面談内容を含まない匿名サマリ（data/analytics/<idHash>.json）です。元の面談を完全削除しても統計だけは長期保存できます。

不採用理由の内訳のみ、生の SessionMeta（data/sessions/<id>/session.json）から集計されます。こちらもセッションを完全削除すると消えます。

横断比較

/compare で複数の評価済セッションを並べて比較します。

画面

選んだ人数によって表示レイアウトが 2 種類あります。

2〜6 件：標準ビュー（候補者を列で並べる）

7 件以上：転置ビュー（候補者を行で並べる）

用途

  • 同じ役割で複数候補者を比較 → どの候補者を採用するか
  • 複数役割を横断比較 → 部署横断採用の意思決定
  • 大人数の候補者から絞り込みたいときは 7 件以上選択して転置ビューでソート活用

Excel エクスポート

マスタ・候補者一覧を Excel ファイルとして自動的に生成します。

自動更新ファイル

  • data/exports/マスタ.xlsx — 求人情報 + 評価条件

FAQ

よく寄せられる質問。

質問一覧

役割を間違えて作成してしまった
一覧画面で対象セッションの 「複製」 をクリックし、複製ダイアログで役割を変更してください。① 候補者情報は引き継がれ、②③ は新役割向けに自動でクリアされます。
③ 質問エリアに何も入力できない
② 条件凍結 を先に行う必要があります。② で「条件を凍結する」をクリックしてから ③ に進んでください。
削除した面談を元に戻したい
ヘッダの 「ゴミ箱」（/trash）から復元できます。猶予期間（既定 7 日）を過ぎると完全削除されます。
バックアップはどう取る？
/settings 下部の「バックアップ」で暗号化パスワードを入れて作成。AES-256-GCM（鍵導出は PBKDF2-SHA256 200k）で暗号化されたアーカイブが data/_backups/ に保存されます。パスワードを忘れると復号不可。
複数 PC で同じデータを使いたい
設定で dataRoot を共有フォルダ（OneDrive 等）に指定すれば共有可能。ただし 同時編集はサポートしません（最後に保存した方が勝つ）。
過去の合格率や軸別スコアを見たい
分析画面 で集計表示されます。匿名化されているので個人特定はできません。
⑤ 総合スコアはどうやって計算されている？（計算式・ベースの詳細）

総合スコアは 6 小軸のスコア × 小軸重みの重み付き平均で算出されます。全ユーザーで同じ計算式・同じベースが使われるよう、以下の 3 レイヤ構造になっています。

ベース構造（2 大分類 × 3 小軸 = 6 軸）

大分類小軸既定重み
人間性主体性3
コミュニケーション力3
学習意欲3
技術力専門知識3
問題解決力3
設計力3

スケールは 0〜5、0.5 刻み、11 段階。ベースは data/master/eval_criteria.json に定義され、/master 画面から編集します。

計算フロー

役割別の重み上書き（ロール別 セクション）

役割ごとに「見たい観点」が違うので、小軸重みと合格/普通ラインを上書きできます。例：

役割主体性コミュ学習専門問題解決設計合格ライン
Dev3344544.0
NW4335544.0
ITSupport3532313.5
PMO5522444.5

ITSupport は「コミュ重み 5・設計力 1」で人間性寄り、PMO は「主体性・コミュ 各 5・合格ライン 4.5」で高い水準を要求、という具合に 役割ごとに評価軸のバランスと合否閾値が変わります。

② 条件凍結による「スナップショット固定」

②「求める人材条件」で 「条件を凍結する」 ボタンを押すと、その時点の役割マスタ＋評価条件（重みも合格ラインも全部）が conditions_snapshot.json に保存されます。以降その面談の総合スコアは、凍結時点のベースで計算されます。

これにより、面談 3 か月後にマスタを触っても過去の面談スコアはブレません。逆に、まだ凍結していない面談は現在マスタで計算されるため、マスタ変更時は要注意です。

全ユーザーで同じスコアを出すには

  • eval_criteria.json を統一する：/master の「エクスポート」で JSON を配布 → 各ユーザーが「インポート」で同じ内容に揃える
  • 役割マスタも統一する：役割別の重み上書きは eval_criteria.json の ロール別 セクション。マスタ配布に含まれる
  • 凍結タイミングを合わせる：マスタ更新前に凍結された面談は旧マスタで計算される。新旧の混在を避けたい場合は、全面談を凍結してからマスタを更新
  • 小軸スコア（0〜5）の付与基準を BARS で明文化：計算式は同じでも、そもそもの点数付けが人によってズレるとスコアもズレる。BARS（Behaviorally Anchored Rating Scale）の水準表を /master の評価条件本文に書いて共通化

計算式そのものはコード内の weightedAverage()（app/sessions/[id]/actions.ts）で固定されているため、ここは全ユーザー共通です。ブレる要因は「ベース（重み・ライン）」と「小軸スコアの付与基準」の 2 点だけ。

/cost の金額はどうやって計算されている？（計算式の詳細）

/cost の金額は、各 API 呼び出しについて以下の 4 ステップで概算しています。実額は Console／請求書を正としますが、感覚を掴む用途には十分な精度です。

ステップ 1  文字数 → token

tokens = ceil(chars ÷ 2.5)。日本語の「1 token ≈ 2.5 文字」を採用（英語は 4 文字前後、混在の妥協値）。実際の tokenizer と差が出るため 上限見積りとして扱ってください。

ステップ 2  token × 単価 → USD

USD = tokens ÷ 1,000,000 × 単価。単価はモデル別・入出力別に設定。lib/pricing.ts の MODEL_PRICING テーブルが正本：

モデル入力 (per 1M tok)出力 (per 1M tok)
Claude Haiku 4.5$1.00$5.00
Claude Sonnet 4.6（既定）$3.00$15.00
Claude Opus 4.7$5.00$25.00
GPT-5.4$2.50$15.00
Gemini 3.5 Flash$1.50$9.00

ステップ 3  入力 + 出力 → 合計 USD

totalUsd = inputUsd + outputUsd。上図では $0.012 + $0.036 = $0.048。

ステップ 4  USD → JPY 換算

JPY = totalUsd × 162。為替は USD $1 = ¥162 で固定（lib/pricing.ts の USD_TO_JPY）。円安局面では実額より少なめに、円高局面では多めに出ます。

実際の /cost 画面での見え方

算入されないもの（＝ 見積りが多めに出る要因）

  • Prompt caching 割引：Anthropic prompt cache のヒット時は入力単価が 10 % / 25 % に下がるが、本ツールは未計算
  • Batch API 割引：Batch モードは 50 % 割引だが、本ツールは同期呼び出し前提
  • tokenizer 精度：実 tokenizer より粗い「文字数 ÷ 2.5」で概算

そのため /cost の表示は「これ以下に収まる」上限見積りと読むのが安全です。

データは PC のどこに保存されている？
既定では data/ フォルダ配下（求人情報・面談セッション・監査ログ・バックアップ）。保存先を変えたい場合は /settings の 「データ保存先」 で任意フォルダを指定できます。OneDrive など同期フォルダに変えれば別 PC からも同じデータを開けます。
履歴書が PDF や Word だがそのまま貼れない
貼付モードでは テキスト しか受け付けません。運用としては、Claude や ChatGPT に PDF/Word を投げて 「経歴サマリ / 主要スキル / 強み」 の 3 見出しで要約させ、その結果を ① に貼り付けます。① 候補者情報を貼る にサンプル形式があります。
一度保存した評価やスコアを後から修正できる？
はい、⑤ 評価セクションを開いて上書き保存すれば evaluation.json と一覧の総合スコア・採否 が書き換わります。採否（採用 / 不採用 / 未確定）の変更も同じ手順です。
ただし 分析画面 の集計は保存期間スイープが匿名サマリを書き出したタイミングのスナップショットなので、日常の上書き保存では更新されません（スイープ後は ⑤ の再編集自体ができなくなります）。
面談者が増えて一覧から探しにくい
一覧画面上部のフィルタで 役割 / 状態 / 採否 / 氏名（部分一致） による絞込みが可能。表ヘッダをクリックすれば作成日やスコア順の並び替えもできます。

トラブル時

よくある問題と対処。

ブラウザが自動で開かない

手動で http://localhost:3939 を開いてください。それでも繋がらない場合：

  • コマンドプロンプトに「Ready」と表示されたか確認
  • ファイアウォール / セキュリティソフトが node.exe をブロックしていないか
  • ポート 3939 が他で使われていないか（netstat -ano | findstr :3939）

Error: spawn EINVAL が出て起動しない

Node.js 22 系以降で next-with-port.mjs が .cmd を spawn するときに発生します（CVE-2024-27980 対策の仕様変更）。
scripts/next-with-port.mjs の spawn オプションに shell: isWin が含まれていることを確認してください。含まれていない場合は追記して保存 → 再度 start-app.bat。

「保存」を押しても反映されない

  • 画面右下に エラーバナーが出ていないか確認
  • F5 でページをリロードして「最終保存：…」のタイムスタンプが更新されたか確認
  • それでもダメなら 監査ログ で該当操作の有無を確認
  • 監査ログにも無ければ、コマンドプロンプトのエラー表示を確認

アプリを完全に止めたい

起動時に開いた コマンドプロンプトで Ctrl+C → Y → Enter。ブラウザを閉じるだけでは停止しません。

データが消えた

  1. まず ゴミ箱 を確認（猶予期間内なら復元可）
  2. data/_backups/ にバックアップがあれば CLI で復号 → 手動配置
  3. OneDrive 等で同期しているなら、サービス側の「履歴」から復元
